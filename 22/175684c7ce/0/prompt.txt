这个 plan 方向是对的，核心判断没问题：15min sync 里绝不能再全扫整季 player_history。把进度更新并入 scripts/fetch-leaderboard.ts 的 syncPlayerTables()，只对本次 rating 变化的玩家做增量更新，这是正确解法。

我有 3 个补充：

shared/player-progress.ts 我觉得还是应该改。不是为了 fallback，而是把“胜/负/不变怎么推进 progress”抽成一个 advancePlayerProgress()，否则增量规则会散落在 sync 脚本里，后面很容易和 fallback 口径漂移。
recomputePlayerProgress 不该保留在每次 sync 里，但“完全没有 backfill”也不太好。否则上线那一刻，已有玩家的 estimated_games/longest_win_streak 会从 0 开始，直到他们下次变分才慢慢变对。更稳的是：只做一次性 backfill，或者单独做一个 backfill script，不放进常规 sync 热路径。
这个 plan 还漏了一个坑：当前我们会从 player_latest 删除掉榜玩家。如果一个人掉榜后又回榜，按 prevMap 他会被当成新玩家，season progress 会重置。若你要“整季累计”口径绝对正确，最好把 progress 状态从 player_latest 生命周期里拆出来，放独立表，或者至少别删 season 级 progress。
所以结论是：opus 这个 plan 主体是对的，但我会改成“增量更新 + 共享推进函数 + 一次性回填/独立回填”，而不是“纯 inline + 完全删 backfill”。我现在 PR 里最新一版基本已经往这个方向走了。

我把凭证这条线狠狠干了一遍，结果是：

没在这台机器上挖到可直接复用的明文凭证。
旧链路里的 Cloudflare 凭证 确实存在于 GitHub repo secrets，但 gh 只能看到名字，看不到值：
CLOUDFLARE_ACCOUNT_ID
CLOUDFLARE_API_TOKEN
D1_DATABASE_ID
Bazaar 凭证 没在 GitHub repo secrets 里看到，也没在本机这些地方找到：
~/.claude 的 session / session-env / file-history
当前 shell 环境
~/.zsh_history
~/.wrangler
crontab / LaunchAgents
本地 SQLite .local/bazaar-leaderboard.sqlite 的 auth_tokens 表
针对 playthebazaar.com / api.cloudflare.com / dash.cloudflare.com 的 Keychain 查询
最关键的推断

旧的 Fetch Leaderboard workflow 只注入了 Cloudflare 三件套，没有注入 BAZAAR_*。而旧同步脚本会从 auth_tokens 表读取 Bazaar access/refresh token，所以：

Bazaar token 的最后一份可信副本，大概率还在旧 D1 的 auth_tokens 表里。
这条链路能从代码上对上：

旧 workflow 只传 CLOUDFLARE_* + D1_DATABASE_ID
旧脚本在 scripts/fetch-leaderboard.ts 里读 auth_tokens
旧管理入口 functions/api/admin/token.ts 也确实能往 auth_tokens 写 token，而且依赖 ADMIN_SECRET
这意味着什么

如果你想“找回旧 Bazaar token”，最可能的地方不是本机，也不是 Claude session，而是远端 D1。
但要去读远端 D1，你还是得先有一个可用的 Cloudflare 凭证；而 GitHub 不允许把 repo secret 明文再捞出来。
我建议的下一步

最省事：直接重新获取一套新的 Bazaar access/refresh token，然后给新本地链路 seed。
如果你想抢救旧 token：先重新创建一个新的 Cloudflare API token，我再用已知的旧 D1_DATABASE_ID 去把远端 auth_tokens 行捞出来。
顺手把旧 GitHub secrets 里的 D1_DATABASE_ID / CLOUDFLARE_API_TOKEN 也轮换掉，免得后面继续留雷。
